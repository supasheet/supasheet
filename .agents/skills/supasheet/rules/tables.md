# Tables

## Order of operations (non-negotiable)

1. `CREATE TABLE` with column definitions and constraints.
2. `COMMENT ON TABLE` (JSON metadata — see `references/table-metadata.md`).
3. `REVOKE ALL` from `public, anon, authenticated, service_role`.
4. `GRANT` exactly the intended operations directly to the specific native
   roles that should hold them.
5. `ENABLE ROW LEVEL SECURITY` + one policy per operation.
6. Indexes.
7. `select supasheet.refresh_metadata();` at the end of the migration.

## 1. Table definition

- PK: `id uuid primary key default extensions.uuid_generate_v4 ()` (or `bigint generated by default as identity`).
- Ownership: `user_id uuid default auth.uid () references supasheet.users (id) on delete set null` — always `supasheet.users`, never `auth.users`.
- Timestamps: `created_at timestamptz default current_timestamp`, `updated_at timestamptz default current_timestamp` (maintain via `supasheet.set_updated_at()` trigger if desired).
- Use `supasheet.*` domain types for rich columns (see `references/data-types.md`).
- Generated columns are fine: `line_total numeric(12,2) generated always as (quantity * unit_price) stored`.
- Constraints inline: `unique (project_id, member_id)`, `check (...)`.
- FK `on delete` semantics: `cascade` for children/junctions, `set null` for optional refs, `restrict` for protected refs (e.g. invoices → clients).

## 2–4. Grants

```sql
revoke all on table app.tickets
from
  public,
  anon,
  authenticated,
  service_role;

grant
select
,
  insert,
update,
delete on table app.tickets to "x-admin";

grant
select
,
  insert,
update on table app.tickets to "user";
```

- `authenticated` never holds a direct grant — every role that needs access
  is granted explicitly, spelled out per role (grants are non-inheriting by
  convention; don't rely on one role being a superset of another).
- Junction tables: `select, insert, delete` (no update).
- Singletons: `select, insert, update` (no delete).
- Column-level grants are supported when only some columns should be writable:
  `grant update (status, assignee_id) on app.tickets to "user";`

## 5. RLS policies

One policy per operation, named `<table>_<action>` (demo style) or `<table>_<action>_policy` (platform style) — pick one and stay consistent. Grants already decided who can attempt the operation, so most policies are just `using (true)`:

```sql
alter table app.tickets enable row level security;

create policy tickets_select on app.tickets for
select
  to authenticated using (true);

create policy tickets_insert on app.tickets for insert to authenticated
with
  check (true);

create policy tickets_update on app.tickets for
update to authenticated using (true)
with
  check (true);

create policy tickets_delete on app.tickets for delete to authenticated using (true);
```

Owner-scoped rows: `using (user_id = auth.uid ())`. An admin-style override
that needs to see rows an ownership check would otherwise hide adds a second
permissive policy: `using (pg_has_role (current_user, 'x-admin', 'member'))`.
See `rules/policies.md` for clause rules.

## 6. Indexes

Index every FK column, every enum/status column used in filters, and the default sort column:

```sql
create index idx_app_tickets_user_id on app.tickets (user_id);

create index idx_app_tickets_status on app.tickets (status);

create index idx_app_tickets_created_at on app.tickets (created_at desc);
```

Explicit method when non-default: `create index ... using gin (metadata);`.

## Altering tables

- `alter table app.tickets add column priority app.priority default 'medium';` then comment + refresh metadata.
- Adding a FK later: `alter table app.tickets add constraint tickets_project_id_fkey foreign key (project_id) references app.projects (id) on delete cascade;`
- Any DDL or comment change requires `select supasheet.refresh_metadata();` to show up in the UI.
